Nomes de Diretório

É comum encontrar nomes de sub diretórios que indicam a presença de uma tecnologia associada. Por exemplo:

servlet - servlets Java pls - gateway Oracle Application Server PL / SQL cfdocs ou cfide - Cold Fusion SilverStream - o servidor da web SilverStream WebObjects ou {function} .woa - Apple WebObjects rails - Ruby on Rails

Tokens de sessão

Muitos servidores da web e plataformas de aplicativos da web geram tokens de sessão por padrão com nomes que fornecem informações sobre a tecnologia em uso. Por exemplo:

JSESSIONID - A plataforma Java ASP SESSIONID - servidor Microsoft IIS ASP.NET_SessionId - Microsoft ASP.NET CFID / CFTOKEN - Cold Fusion PHPSESSID - PHP

Componentes de código de terceiros

Muitos aplicativos da web incorporam componentes de código de terceiros para implementar funcionalidades comuns, como carrinhos de compras, mecanismos de login e painéis de mensagens. Eles podem ser de código aberto ou podem ter sido adquiridos de um desenvolvedor de software externo. Quando for esse o caso, os mesmos componentes geralmente aparecem em vários outros aplicativos da web na Internet, que você pode inspecionar para entender como o componente funciona. Freqüentemente, outros aplicativos usam recursos diferentes do mesmo componente, permitindo que você identifique comportamento e funcionalidade adicionais além do que está diretamente visível no aplicativo de destino. Além disso, o software pode conter vulnerabilidades conhecidas que foram discutidas em outro lugar, ou você pode baixar e instalar o componente por conta própria e realizar uma revisão do código-fonte ou sondá-lo em busca de defeitos de maneira controlada.

1. Identifique todos os pontos de entrada para entrada do usuário, incluindo URLs, parâmetros de string de consulta, dados POST, cookies e outros cabeçalhos HTTP processados ​​pelo aplicativo.

2. Examine o formato da string de consulta usado pelo aplicativo. Se não utilizar o formato padrão descrito no Capítulo 3, tente entender como os parâmetros estão sendo transmitidos por meio do URL. Praticamente todos personalizados esquemas ainda empregam alguma variação no modelo de nome / valor, então tente entender como os pares nome / valor estão sendo encapsulados nos URLs não padrão que você identificou.

3. Identifique quaisquer canais fora do limite por meio dos quais dados controláveis ​​pelo usuário ou outros dados de terceiros estão sendo introduzidos no processamento do aplicativo.

4. Visualize o banner do servidor HTTP retornado pelo aplicativo. Observe que, em alguns casos, diferentes áreas do aplicativo são tratadas por diferentes componentes de back-end, portanto, diferentes cabeçalhos de servidor podem ser recebidos.

6. Verifique se há outros identificadores de software contidos em quaisquer cabeçalhos HTTP personalizados ou comentários de código-fonte HTML.

7. Execute a ferramenta httprint para fazer a impressão digital do servidor da web.

8. Se informações refinadas forem obtidas sobre o servidor da web e outros componentes, pesquise as versões de software em uso para identificar quaisquer vulnerabilidades que possam ser exploradas para avançar um ataque (consulte o Capítulo 18).

9. Revise seu mapa de URLs de aplicativos para identificar quaisquer extensões de arquivo de aparência interessante, diretórios ou outras subseqüências que possam fornecer pistas sobre as tecnologias em uso no servidor.

10. Revise os nomes de todos os tokens de sessão emitidos pelo aplicativo para identificar as tecnologias que estão sendo usadas.

11. Use listas de tecnologias comuns, ou Google, para estabelecer quais tecnologias podem estar em uso no servidor, ou descobrir outros sites e aplicativos que parecem empregar as mesmas tecnologias.

12. Faça pesquisas no Google em busca de nomes de cookies, scripts, cabeçalhos HTTP incomuns e semelhantes que possam pertencer a componentes de software de terceiros. Se você localizar outros aplicativos nos quais os mesmos componentes estão sendo usados, revise-os para identificar qualquer funcionalidade e parâmetros adicionais que os componentes suportam e verifique se eles também estão presentes em seu aplicativo de destino. Observe que os componentes de terceiros podem ter uma aparência e comportamento bastante diferentes em cada implementação, devido às personalizações de marca, mas a funcionalidade principal, incluindo nomes de script e parâmetro, geralmente é a mesma. Se possível, baixe e instale o componente e analise-o para compreender totalmente seus recursos e, se possível, descobrir quaisquer vulnerabilidades. Consulte repositórios de vulnerabilidades conhecidas para identificar quaisquer defeitos conhecidos com o componente em questão.

Identificação da funcionalidade do lado do servidor

Freqüentemente, é possível inferir muito sobre a funcionalidade e a estrutura do lado do servidor, ou pelo menos fazer uma suposição fundamentada, observando as pistas que o aplicativo divulga ao cliente.

Como você viu, a extensão do arquivo .jsp indica que as páginas do servidor Java estão em uso.

Você pode supor que uma função de pesquisa recuperará suas informações de um sistema de indexação ou de um banco de dados. A presença do parâmetro OrderBy sugere que um banco de dados back-end está sendo usado e que o valor enviado pode ser usado como a cláusula ORDER BY de uma consulta SQL. Esse parâmetro pode ser vulnerável à injeção de SQL, assim como qualquer um dos outros parâmetros se forem usados ​​em consultas de banco de dados (consulte o Capítulo 9).

Entre os outros parâmetros, também é interessante o campo isExpired. Parece ser um sinalizador booleano que especifica se a consulta de pesquisa deve incluir conteúdo expirado. Se os designers do aplicativo não esperassem que os usuários comuns pudessem recuperar qualquer conteúdo expirado, alterar este parâmetro de 0 para 1 poderia identificar uma vulnerabilidade de controle de acesso (consulte o Capítulo 8).

É possível que o aplicativo recupere o arquivo de modelo especificado e inclua o conteúdo em sua resposta. Esses parâmetros podem ser vulneráveis ​​a ataques de travessia de caminho, permitindo que arquivos arbitrários sejam lidos do servidor (consulte o Capítulo 10).

Também é interessante o parâmetro de edição, que é definido como falso. Pode ser que a alteração desse valor para verdadeiro modifique a funcionalidade de registro, potencialmente permitindo que um invasor edite itens que o desenvolvedor do aplicativo não pretendia que fossem editáveis. O parâmetro ver não tem nenhuma finalidade facilmente adivinhada, mas pode ser que modificá-lo fará com que o aplicativo execute um conjunto diferente de funções que um invasor poderia explorar.
